What is this endpoint?

/api/stats is a custom runtime endpoint that exposes live server-side data: how long the app has been running, how many requests it has handled (tracked with a simple in-memory counter via @app.before_request), the hostname, Python version, and environment. Useful for observability without needing Prometheus.

📄 Environment Variables — GET /api/env
Safe Environment Variables 0 vars

What is this endpoint?

/api/env returns environment variables filtered to safe prefixes only (APP_, FLASK_, HOSTNAME, PATH, etc.). Secret keys, database URLs, and tokens are never exposed. This is how Kubernetes ConfigMaps and environment-based config are typically inspected at runtime — you can see exactly what the running container sees without SSHing into it.

What is this endpoint?

POST /api/echo accepts any JSON body and echoes it back with metadata: timestamp, content-type, and method. It's useful for testing that your HTTP client, proxy, or ingress is forwarding request bodies correctly. In a real pipeline you might use this to verify that a webhook payload survives transit through an API gateway unchanged.

Liveness answers: "Is the process alive?" If this fails, Kubernetes restarts the container. It should never do expensive work — just return 200 instantly.
Readiness Probe GET /ready
Status—
HTTP Code—
Uptime—
Readiness answers: "Is the app ready to serve traffic?" If this fails, Kubernetes removes the pod from the Service (no restart). Returns 503 for the first 2 seconds after startup.
Liveness vs Readiness — when to use each
✓ Liveness (/health)
  • App process is alive
  • Not deadlocked
  • Always fast (no DB check)
  • Failure → container restart
✓ Readiness (/ready)
  • Warmed up and ready
  • Can check DB / cache
  • Failure → removed from LB
  • No restart on failure

📚 Project Explanation

What is this project?

A minimal Flask web application wired into a complete DevOps toolchain. It demonstrates the full lifecycle of a production-grade service — from writing code, to testing, containerising, scanning, deploying, and monitoring — all running locally with no cloud account required.

Flask Docker Kubernetes Terraform GitHub Actions Prometheus Grafana Trivy

The Flask App (app.py)

Four routes: / returns a welcome message with version and environment. /health is the liveness probe — returns {"status":"ok"}. /info returns the hostname (useful in Kubernetes to see which pod responded). /metrics is auto-instrumented by prometheus-flask-exporter — no manual metric code needed.

Version and environment are read from APP_VERSION and APP_ENV environment variables, so the same image works in dev, staging, and production just by changing env vars.

Docker & Multi-stage Build

The Dockerfile uses a multi-stage build on python:3.11-slim to keep the image small. The app runs as a non-root user for security. A HEALTHCHECK instruction tells Docker to poll /health periodically — if it fails, Docker marks the container unhealthy and can restart it.

Terraform (Infrastructure as Code)

Uses the kreuzwerker/docker provider to provision a Docker container, named network, and volume — all locally. Key variables like app_name, container_port, and image_tag are overridable at terraform apply time. This means you can spin up or destroy the whole stack with one command.

Kubernetes Manifests (k8s/)

Production-ready manifests in the flask-app namespace. The Deployment runs 2 replicas with resource requests/limits and both liveness and readiness probes. The HPA (Horizontal Pod Autoscaler) scales from 2 to 5 pods when CPU exceeds 70% or memory exceeds 80%. A ClusterIP Service load-balances across pods, and an Ingress exposes it via Nginx with a TLS placeholder.

CI/CD Pipeline (GitHub Actions)

Five sequential jobs — each needs: the previous so a failing test blocks the build. The Docker image is pushed to GHCR tagged with the short git SHA, making every build traceable. Trivy scans the image for CVEs and uploads a SARIF report to the GitHub Security tab. The deploy job only runs on main branch pushes and is currently simulated (would be kubectl set image in production).

Monitoring (Prometheus + Grafana)

Prometheus scrapes /metrics every 15 seconds. Grafana connects to Prometheus at http://prometheus:9090. Both run in a separate Compose file (monitoring/docker-compose.monitoring.yml) so they're independent from the app stack. The recommended Grafana dashboard is ID 11074 (Flask Requests). All metrics are auto-generated — request counts, durations, and Python process stats.

Why no cloud required?

Terraform uses a local Docker provider instead of AWS/GCP/Azure. Kubernetes manifests target a local cluster (Docker Desktop, minikube, or kind). The monitoring stack runs in Docker Compose. GitHub Actions is the only external dependency — and even that only runs when you push to GitHub. This design makes the project a self-contained portfolio piece anyone can run on their laptop.
